A README is often the first file someone reads in a project. It should explain what the project does, how to install it, how to run it, and where to start.

You do not need a perfect README on the first try. A structured draft is enough to help future you and other developers understand the project.

## What a good README includes

- Project name.
- Short description.
- Tech stack.
- Installation command.
- Run or development command.
- Key features.
- Usage notes.
- License or contribution notes when relevant.

The [README Generator](/tools/readme-generator) helps you turn those basics into a Markdown file quickly.

## Example README structure

```md
# Project Name

Short project description.

## Tech Stack

- Next.js
- TypeScript
- Tailwind CSS

## Installation

npm install

## Usage

npm run dev
```

## Step-by-step README workflow

1. Write a one-sentence project description.
2. List the main technologies.
3. Add install and run commands.
4. Add the most important features.
5. Generate a draft.
6. Edit the draft with project-specific details.

## Common README mistakes

- Describing the code but not how to run it.
- Forgetting environment variable setup.
- Leaving placeholder text in the final README.
- Making the README too long before the project is stable.

## FAQ

### Should every project have a README?

Yes. Even small projects benefit from a short README with setup and usage instructions.

### Can a README generator replace editing?

No. It gives you a draft. You should still review and customize the details.

### Should I include screenshots?

Screenshots are helpful for UI projects, but setup instructions matter first.
